# System Design Document

## 1. Caching Strategy

Cache invalidation is handled using the **cache-aside pattern** with targeted key deletion. When a write event occurs (task created, member added, etc.), we explicitly delete the affected cache keys rather than updating them — on the next read, a cache miss triggers a fresh database fetch which then re-populates the cache. This keeps the cache consistent without requiring distributed transactions. If the cache and database fall out of sync (e.g., a Redis crash between a write and invalidation), the TTL acts as the final safety net: `projects:user:{userId}` keys expire after 5 minutes and `project:{projectId}` keys after 2 minutes, bounding the maximum staleness window. The limits of this approach are that during the TTL window a client may briefly see stale data, and under high write contention (many concurrent task updates) the invalidation fan-out can cause a brief "thundering herd" as all clients miss the cache simultaneously; for production, a short jitter on TTL and a queue-based invalidation debounce would mitigate this.

## 2. Failure Handling

If an export job fails mid-execution (e.g., database query throws, disk write fails), the BullMQ worker's `failed` event is fired. BullMQ is configured with `attempts: 3` and an **exponential backoff** (`type: 'exponential', delay: 2000ms`), meaning retries wait 2s, 4s, then 8s before each attempt. Only after all three attempts are exhausted does the `failed` handler mark the `exports` record as `status = 'failed'`. Partial CSV files are never committed to disk or referenced in the database until the worker explicitly writes the completed path. To make failures more observable in production, I would add: (1) a structured logger (Pino/Winston) emitting JSON logs so failures are searchable in a log aggregator, (2) an alerting webhook or Slack notification on permanent job failure, and (3) a BullMQ Bull-Board dashboard for real-time queue visibility.

## 3. Access Control

RBAC is enforced in the **service layer**, not controllers or middleware. The controller extracts `req.user.id` from the verified JWT and passes it to the service. The service then queries `ProjectMember` to confirm the user is either an owner or member before returning data, and checks `Project.ownerId` for owner-only operations (delete task, add/remove members, trigger export). Returning `403` instead of `404` for inaccessible resources is a deliberate choice — a `404` would implicitly leak whether the resource exists, while a `403` communicates "you are authenticated but not authorized" without revealing existence. Keeping the check in the service layer means it is impossible to bypass via route reordering or adding new routes — every code path that touches data goes through the same guard.

## 4. Scale

At 10,000 concurrent users, the first two bottlenecks I would address are: **(1) Database connection pool exhaustion** — by default PostgreSQL allows ~100 connections, and each Node.js process holds open connections via Prisma. The fix is to introduce **PgBouncer** as a connection pooler in transaction mode, allowing thousands of app-level "connections" to share a small pool of real DB connections. **(2) Redis single-instance saturation** — all caching, queuing, and refresh token lookups hit one Redis node. Under high load, this becomes a hotspot. I would migrate to a **Redis Cluster** (at least 3 shards), shard cache keys by user/project ID, and consider separating the BullMQ queue Redis instance from the cache Redis instance so queue throughput does not affect cache latency.
